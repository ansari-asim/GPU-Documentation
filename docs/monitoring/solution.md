# Solution Monitoring

Monitors Python scripts, .NET apps, Linux systemd services, and IIS sites. Sends 🔴 Down and 🟢 Recovered alerts with downtime duration. Persists state across restarts via `service_state.json`.

[:material-download: Download `solution_monitor.py`](../scripts/solution_monitor.py){ .md-button download="solution_monitor.py" }

---

## How It Works

Each service type is detected differently:

**Python scripts** — `psutil` scans all running process command lines for `process_name`. Only counted as running if RSS memory > 0 and status is not zombie/dead/stopped.

**dotnet apps** — same `psutil` command-line scan, matching the `.dll` name in process arguments.

**Linux systemd services** — runs `systemctl is-active <service>` and checks for `active`.

**IIS sites** — runs `Get-Website` via PowerShell on Windows and checks for `Started`.

All checks run in parallel. State is saved to `service_state.json` after every cycle so restarts don't fire duplicate DOWN alerts for already-known outages.

## Configuration

```json title="config.json"
{
  "check_interval_seconds": 600,

  "python_scripts": [
    { "name": "Solution", "process_name": "my_script.py", "enabled": true }
  ],

  "dotnet_apps": [
    { "name": "Dashboard", "process_name": "MyApp.dll", "enabled": false }
  ],

  "linux_services": [
    { "name": "Dashboard",      "service_name": "dashboard.service",      "enabled": true },
    { "name": "ANPR_API",       "service_name": "api.service",            "enabled": true },
    { "name": "Camera_Monitor", "service_name": "camera-monitor.service", "enabled": true }
  ],

  "iis_sites": [
    { "name": "Customer API", "site_name": "CustomerApiPool", "enabled": false }
  ],

  "email": {
    "enabled": true,
    "from_email": "dev@example.com",
    "to_emails": ["admin@example.com"],
    "subject_down": "Service DOWN Alert",
    "subject_recovered": "Service RECOVERED Alert",
    "body_down": "{service_name} has stopped running.",
    "body_recovered": "{service_name} is running again."
  },

  "google_chat_enabled": true,
  "state_file": "service_state.json"
}
```

| Key | Type | Description |
|---|---|---|
| `check_interval_seconds` | int | Poll frequency in seconds |
| `python_scripts` | list | Python processes — matched by command line |
| `dotnet_apps` | list | .NET processes — matched by `.dll` name |
| `linux_services` | list | systemd services checked via `systemctl` |
| `iis_sites` | list | IIS app pools checked via PowerShell |
| `email.body_down` | string | Supports `{service_name}` placeholder |
| `email.body_recovered` | string | Supports `{service_name}` placeholder |
| `google_chat_enabled` | bool | Enable Google Chat alerts |
| `state_file` | string | Path to persisted state JSON |

Set `"enabled": false` on any entry to skip it without removing it from config.

---

## State File

`service_state.json` is written automatically after every poll. Keys follow the format `<type>::<name>`:

```json title="service_state.json (example)"
{
  "linux::Dashboard":      true,
  "linux::ANPR_API":       true,
  "linux::Camera_Monitor": false
}
```

Do not edit this file manually. Delete it to reset alert state.
